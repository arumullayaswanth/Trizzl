1 What We Are Building

Trizzl is a borderless social fintech wallet that lets people send money, hold multi-currency balances, escrow funds, split bills, chat and pay, use debit cards, and get AI financial guidance. It is designed for the diaspora, immigrants, international students, and anyone living across borders.

The regulated banking infrastructure is powered by DC Bank API, which handles the actual money movement, card issuance, and compliance rails. Trizzl's backend is the orchestration layer that sits on top — handling business logic, user experience, and product features.

📌 Architecture Pattern: Domain-Driven Microservices Each of the 12 services owns its own database, its own codebase, and its own deployment. No service ever reads directly from another service's database — data sharing happens only via REST API calls (synchronous) or RabbitMQ events (asynchronous). This is how Stripe, Monzo, and N26 are built.

// OTP Generation and Storage (Redis, 5-minute TTL)
func (s *AuthService) SendOTP(ctx context.Context, phone string) error {
    otp := fmt.Sprintf("%06d", rand.Intn(1000000))
    key := fmt.Sprintf("otp:%s", phone)
    // Store in Redis with 5-minute TTL, max 3 attempts
    err := s.redis.Set(ctx, key, otp, 5*time.Minute).Err()
    if err != nil { return err }
    return s.twilio.SendSMS(phone, fmt.Sprintf("Your Trizzl code: %s", otp))
}

// JWT Token Pair — RS256 asymmetric signing

9 Service 3: Wallet Service Phase 1

The financial ledger of Trizzl. Manages all user balances, FlashWallet creation, multi-currency accounts, funding (top-up), withdrawals, and complete transaction history.

🚨 CRITICAL RULE: Double-Entry Bookkeeping — Non-Negotiable Every single balance change MUST create two ledger entries: one DEBIT and one CREDIT. You must never directly UPDATE a balance column. This is the foundation of all regulated financial software. Every dollar in must equal every dollar out when you audit the ledger. Failure to implement this correctly is a compliance and audit disaster.

10 Service 4: Payment Service Phase 1

Orchestrates all money movement. TrizzlMe (P2P by TrizzlTag or QR code), FlyFunds (international remittance), CampusRemit (tuition/education payments). Calls DC Bank API for actual fund movement and updates the Wallet Service ledger via RabbitMQ events.

🚨 Idempotency Keys — REQUIRED on every payment endpoint The client must send a unique UUID (X-Idempotency-Key header) with every payment request. The server stores this key in Redis for 24 hours. If the same key is received again (e.g. network retry), return the original result without processing again. This prevents duplicate charges. This is how Stripe, and every serious payments API works.

// Idempotency middleware — must wrap ALL payment handlers
func IdempotencyMiddleware(redis *redis.Client) gin.HandlerFunc {
    return func(c *gin.Context) {
        key := c.GetHeader("X-Idempotency-Key")
        if key == "" {
            c.JSON(400, gin.H{"error": "X-Idempotency-Key header required"})
            c.Abort(); return
        }
        cacheKey := fmt.Sprintf("idempotency:%s", key)
        cached, err := redis.Get(c, cacheKey).Result()
        if err == nil {
            // Already processed — return cached response
            c.JSON(200, json.RawMessage(cached)); c.Abort(); return
        }
        // Store result in Redis for 24 hours after processing
        c.Set("idempotency_key", key)
        c.Next()
    }
}

// FX Rate fetching with Redis caching (60-second TTL)
func (s *FXService) GetRate(ctx context.Context, from, to string) (float64, error) {
    cacheKey := fmt.Sprintf("fx:rate:%s:%s", from, to)

    // 1. Try Redis cache first (60-second TTL)
    cached, err := s.redis.Get(ctx, cacheKey).Float64()
    if err == nil { return cached, nil }

    // 2. Fetch from Open Exchange Rates API
    midRate, err := s.oxrClient.GetRate(from, to)
    if err != nil { return 0, err }

    // 3. Apply Trizzl FX spread (e.g. 1.5% on conversion)
    trizzlRate := midRate * (1 - s.config.FXSpread)

    // 4. Store in cache for 60 seconds
    s.redis.Set(ctx, cacheKey, trizzlRate, 60*time.Second)
    return trizzlRate, nil
}

type Client struct {
    baseURL    string
    apiKey     string
    httpClient *http.Client

// CreateTransfer — called by Payment Service
func (c *Client) CreateTransfer(ctx context.Context, req TransferRequest) (*TransferResponse, error) {
    body, _ := json.Marshal(req)
    httpReq, _ := http.NewRequestWithContext(ctx, http.MethodPost,
        c.baseURL+"/v1/transfers", bytes.NewBuffer(body))
    httpReq.Header.Set("Authorization", "Bearer "+c.apiKey)
    httpReq.Header.Set("Content-Type", "application/json")
    httpReq.Header.Set("X-Idempotency-Key", req.IdempotencyKey)

    resp, err := c.httpClient.Do(httpReq)
    if err != nil { return nil, fmt.Errorf("dcbank transfer: %w", err) }
    defer resp.Body.Close()

    var result TransferResponse
    json.NewDecoder(resp.Body).Decode(&result)
    return &result, nil
}

// Webhook handler — verify signature BEFORE processing
func VerifyDCBankWebhook(secret string, payload []byte, sig string) bool {
    mac := hmac.New(sha256.New, []byte(secret))
    mac.Write(payload)
    expected := hex.EncodeToString(mac.Sum(nil))
    return hmac.Equal([]byte(expected), []byte(sig))
}

func (p *Publisher) PublishPaymentCompleted(ctx context.Context, event PaymentCompletedEvent) error {
    body, _ := json.Marshal(event)
    return p.channel.PublishWithContext(ctx,
        "trizzl.events",                // exchange
        "payment.transaction.completed",  // routing key
        false, false,
        amqp.Publishing{
            ContentType:  "application/json",
            Body:         body,
            DeliveryMode: amqp.Persistent,   // survive RabbitMQ restart
        })
}

func (c *Consumer) ConsumePaymentCompleted() {
    msgs, _ := c.channel.Consume("notify.payment.completed", "", false, false, false, false, nil)
    for msg := range msgs {
        var event PaymentCompletedEvent
        json.Unmarshal(msg.Body, &event)
        c.notifySvc.PushToUser(event.ReceiverID, "You received money!")
        msg.Ack(false)  // Always ACK after successful processing
    }
}

23 Security

Authentication & Tokens

  • JWT signing: RS256 (asymmetric RSA) — private key signs tokens, public key verifies. Never use HS256 in production.
  • Token TTLs: Access token 15 minutes, refresh token 30 days with rotation on every use.
  • Token revocation: Maintain a Redis denylist. On logout, add the token JTI to Redis with TTL matching token expiry.
  • Transaction MFA: Payments above configurable threshold (e.g. CAD $500) require re-confirmation via OTP or biometric.

Data Protection

  • PII encryption at rest: AES-256-GCM for names, phone numbers, addresses in PostgreSQL.
  • All transit: TLS 1.3 enforced, TLS 1.0/1.1 disabled at ALB level.
  • Card data: Trizzl NEVER stores raw PAN, CVV, or full card numbers. Only tokenized references from DC Bank.
  • KYC documents: Stored in S3 with SSE-S3 server-side encryption. Pre-signed URLs expire in 15 minutes.
  • Database: AWS RDS encryption at rest enabled (AES-256), VPC private subnet (not publicly accessible).

API Security

  • Idempotency keys on all mutating payment endpoints (required header).
  • Webhook verification — HMAC-SHA256 signature from DC Bank validated before processing every webhook.
  • IP allowlist for DC Bank webhook source IPs via Kong ip-restriction plugin.
  • Input validation with go-playground/validator on every request struct.
  • SQL injection prevention via GORM parameterized queries — never use raw string concatenation in SQL.
  • Secrets rotation — AWS Secrets Manager with automatic rotation for DB passwords every 30 days.
